# MacCleaner: уведомление о нагреве и высокой нагрузке

Дата анализа: 2026-07-13
Основание для roadmap: `IMP-036`

## Вывод

Функцию имеет смысл добавить. MacCleaner уже получает большую часть необходимых данных, но нагрев и загрузку нельзя считать одним и тем же состоянием:

- высокая CPU/GPU-нагрузка может ещё не привести к высокой температуре;
- температура может оставаться высокой после завершения нагрузки;
- фоновые процессы, зарядка, внешняя температура и охлаждение также влияют на нагрев.

Поэтому уведомление должно классифицировать причину как минимум в два независимых сигнала: `Sustained Load` и `Thermal Warning`. Если оба сигнала активны, их можно объединить в одно уведомление.

## Что уже есть в приложении

| Возможность | Текущее состояние |
| --- | --- |
| Общая загрузка CPU | Уже собирается `SystemMonitor` |
| Температуры CPU/SoC/battery | Уже читаются через `SMCService` и сохраняются в `thermalHistory` |
| Список процессов | Уже собирается `processNodes` |
| Top процессов | Уже формируется `topProcesses`, сортируется по CPU и ограничивается 15 элементами |
| Краткое отображение top процессов | Menu bar popover уже показывает три процесса с CPU и RAM |
| Системные уведомления | Не реализованы: вызовов `UNUserNotificationCenter` в проекте нет |
| Переход из уведомления | Не реализован; вкладка выбирается локальным `@State` и аргументами только при старте приложения |

Основные точки повторного использования:

- `MacCleaner/Services/SystemMonitor.swift`: `refresh()` уже объединяет CPU, thermal и процессы;
- `MacCleaner/MacCleanerApp.swift`: `PopoverTopProcessesView` уже содержит компактное представление top-3;
- `MacCleaner/Views/ContentView.swift`: вкладки `Processes` и `Fans` уже существуют, но нужен общий navigation router для внешнего события.

## Предлагаемое поведение

### Сигналы

1. `High Load`: высокая общая нагрузка сохраняется непрерывно заданное время, например 30–60 секунд.
2. `Thermal Warning`: системный thermal state достигает `serious`/`critical` либо поддерживаемая температура остаётся выше откалиброванного порога.
3. `Heating Trend`: температура быстро растёт одновременно с устойчивой нагрузкой. Использовать только как дополнительный контекст, а не самостоятельную тревогу до калибровки.

Пороговые значения должны быть настройками и проходить проверку на реальных Mac. Одного мгновенного sample недостаточно.

### Содержание уведомления

- причина: высокая нагрузка, нагрев или оба состояния;
- текущая CPU load и доступная температура;
- top-3 процессов по устойчивой CPU-нагрузке;
- действие `Открыть процессы`;
- дополнительное действие `Открыть температуры и вентиляторы` для thermal warning.

Пример:

```text
Mac работает под высокой нагрузкой
CPU 86% · CPU temperature 91 °C
Top: Xcode 145%, Simulator 82%, browser 34%
[Открыть процессы]
```

## Защита от ложных уведомлений

- использовать rolling window, а не одиночную выборку;
- вводить hysteresis: отдельные пороги входа и выхода из warning;
- cooldown одного типа уведомления не менее 10–15 минут;
- не уведомлять повторно, пока состояние не вернулось в normal;
- группировать helper-процессы одного приложения;
- не объявлять конкретный процесс причиной нагрева, если доказана только корреляция по CPU;
- не запускать дополнительный частый SMC/process polling только ради уведомлений без измерения energy cost;
- добавить пользовательские настройки: включение функции, типы сигналов, чувствительность и режим «Не беспокоить».

## Что потребуется реализовать

1. `LoadThermalAlertService` со state machine `normal → pending → alerted → cooldown → normal`.
2. Разрешение и отправка локальных уведомлений через `UNUserNotificationCenter`.
3. Rolling samples CPU, thermal state, temperatures и top processes.
4. Группировка процессов и ranking по устойчивой, а не мгновенной нагрузке.
5. Общий navigation router для открытия главного окна на вкладке `Processes` или `Fans`.
6. Настройки уведомлений и сохранение cooldown state.
7. Unit-тесты thresholds/state machine и UI-тест перехода из notification action.
8. A/B-измерение собственной CPU/energy стоимости фонового мониторинга.

## Критерии готовности

- краткий всплеск CPU не создаёт уведомление;
- sustained load создаёт не более одного уведомления до восстановления;
- thermal warning работает даже без процесса, который можно уверенно назвать причиной;
- уведомление показывает до трёх актуальных процессов;
- нажатие открывает приложение сразу в нужном разделе;
- при отсутствии разрешения функция объясняет, как включить notifications;
- отключение функции прекращает дополнительный мониторинг;
- влияние фонового сервиса на CPU, wakeups и energy измерено и включено в итоговый отчёт.
