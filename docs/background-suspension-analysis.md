# MacCleaner: приостановка ненужной фоновой работы

Дата анализа: 2026-07-13
Основание для roadmap: `IMP-037`

## Вывод

Функцию следует добавить. Текущая версия уменьшает частоту мониторинга при отсутствии активных экранов, но продолжает выполнять фоновую работу. Нужна централизованная lifecycle-модель, которая при скрытом или свёрнутом главном окне останавливает UI-мониторинг и восстанавливает его при возвращении пользователя.

Приостанавливать следует только работу, не нужную в фоне. Пользовательская очистка, сканирование, обновление или другая явно запущенная операция не должны молча замораживаться.

## Что происходит сейчас

| Область | Текущее поведение без активных consumers |
| --- | --- |
| Общий timer | Продолжает работать каждые 30 секунд |
| CPU, RAM и network | Обновляются на каждом tick |
| Температуры, fans и GPU | Обновляются примерно раз в 2 минуты |
| Процессы и окна | Обновляются примерно раз в 30 минут |
| Disks | Обновляются вместе с процессами |
| Battery | Обновляется примерно раз в 10 минут |
| External battery devices | Тяжёлое обновление ограничено примерно 6 часами |

`SystemMonitor` уже имеет полезную основу:

- `activeConsumers` различает Dashboard, Processes, Windows, Fans и AI;
- активный интервал равен 15 секундам, idle-интервал — 30 секундам;
- тяжёлые источники запускаются реже;
- тест проверяет снижение cadence без consumers.

Но `activeConsumers.isEmpty` означает только переход к idle cadence, а не suspension. Нет единого состояния, связанного с видимостью и минимизацией главного окна.

## Предлагаемая lifecycle-модель

| Состояние | Когда применяется | Разрешённая работа |
| --- | --- | --- |
| `foregroundActive` | Главное окно видимо либо открыт menu bar popover | Полный мониторинг только для видимых consumers |
| `backgroundAlerts` | Окно скрыто, но включены уведомления `IMP-036` | Минимальный alert sampler без UI history и лишних источников |
| `suspended` | Окно скрыто и фоновые функции отключены | Timer и необязательные collectors остановлены |
| `operationRunning` | Пользователь явно запустил scan/cleanup/update | Операция продолжается; UI-анимации и unrelated monitoring остановлены |

Переход нельзя связывать только с `applicationDidResignActive`: окно может оставаться видимым, пока пользователь работает в другом приложении. Нужны события фактической видимости окна: minimize, order out/close, restore и открытие popover.

## Что приостанавливать

- timer полного `SystemMonitor`;
- CPU/RAM/network history, если они не нужны alert sampler;
- SMC polling, fan polling и GPU polling;
- `/bin/ps`, получение окон и top-process ranking;
- disk и battery refresh, кроме отдельно запланированных редких задач;
- animations, preview refresh и диагностические timers невидимых экранов;
- необязательные фоновые проверки UI-моделей.

## Что не приостанавливать автоматически

- уже подтверждённую очистку или перенос в Trash;
- явно запущенный scan, если его контракт предусматривает работу после сворачивания;
- управление вентиляторами: при ручном режиме safety-controller должен продолжать работу либо немедленно вернуть Auto;
- критическую state machine обновления;
- минимальный sampler уведомлений, если пользователь включил `IMP-036`.

## Связь с уведомлениями IMP-036

Полная остановка и фоновые уведомления о нагрузке несовместимы. Поэтому настройка должна давать два понятных варианта:

1. `Pause completely when hidden` — почти нулевая фоновая нагрузка, thermal/load alerts отключены.
2. `Keep lightweight alerts` — работает отдельный редкий sampler с измеренным energy budget.

Alert sampler не должен поддерживать весь `SystemMonitor` активным. Ему нужны только rolling CPU/thermal samples и редкое получение top processes после подтверждения sustained warning.

## Что потребуется реализовать

1. `AppActivityCoordinator` с состояниями foreground/backgroundAlerts/suspended/operationRunning.
2. Отслеживание видимости главного окна и menu bar popover через macOS lifecycle events.
3. Методы `SystemMonitor.suspend()`, `resumeAndRefresh()` и отдельный lightweight alert mode.
4. Реестр долгих операций, запрещающий случайную остановку cleanup/scan.
5. Отмену timers и observation tasks невидимых feature views.
6. Немедленный `force refresh` после восстановления окна без потери выбранной вкладки и состояния UI.
7. Настройки поведения в фоне и явное объяснение компромисса с уведомлениями.
8. Unit-тесты переходов lifecycle и runtime A/B-тест visible/hidden/resumed.

## Критерии готовности

- без фоновых уведомлений после скрытия главного окна отсутствуют периодические `SystemMonitor` ticks;
- SMC, GPU, `/bin/ps`, window и disk collectors не запускаются в suspended;
- menu bar popover временно активирует только необходимые collectors;
- пользовательская операция не прерывается и не дублируется при minimize/restore;
- после restore актуальные данные появляются не позднее одного refresh cycle;
- выбранная вкладка и результаты работы сохраняются;
- в режиме lightweight alerts измерены CPU, wakeups, energy и RSS;
- A/B-отчёт показывает разницу между visible, suspended и backgroundAlerts.

## Метрики результата

| Метрика | До | Цель после изменения |
| --- | ---: | ---: |
| Monitoring timer в suspended | Tick каждые 30 s | 0 |
| CPU/RAM/network history updates | Каждые 30 s | 0 |
| Sensor polling | Примерно каждые 2 min | 0 без alerts |
| Process polling | Примерно каждые 30 min | 0 без alerts |
| Resume latency | Не измерено | Измерить p50/p95 |
| Background CPU/wakeups | Не измерено отдельно | Измерить и сравнить с baseline |

Численные CPU/energy цели следует установить после отдельного baseline-теста свёрнутого приложения.
