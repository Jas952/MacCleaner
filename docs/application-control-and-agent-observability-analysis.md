# Управление приложениями и наблюдаемость AI-агентов

## Цель

Расширить MacCleaner двумя связанными направлениями:

1. дать пользователю визуальную область, куда можно перетащить приложение и запустить контролируемую сессию наблюдения за его нагрузкой;
2. превратить раздел AI Agents из статического обзора установок и процессов в локальный центр наблюдаемости, лимитов, сравнительных тестов и связей агента с другими инструментами.

Оба направления должны использовать один принцип: показывать только измеренные или явно сконфигурированные связи и не выдавать корреляцию за доказанную причину нагрузки.

## Фактическая основа в текущем приложении

- `ProcessTreeService` уже получает процессы, CPU, память, parent PID и поддерживает явные terminate/force-terminate действия.
- `ProcessDetailService` показывает parent chain, число дочерних процессов и открытых файлов.
- `AIWorkloadService` уже строит профили Codex, Hermes, Antigravity, Devin, Claude Code, Cursor, Aider, Goose и Continue.
- Для профиля агента собираются основной процесс, потомки, MCP-процессы, helper-процессы, terminal children, найденные components и skills.
- Текущее сопоставление основано на install anchors, executable paths, command line и ключевых словах. Истории наблюдений, provider quota, арены и временного event graph пока нет.

Это позволяет переиспользовать существующие модели процессов, но долговременная аналитика и атрибуция должны быть отдельными сервисами, а не логикой внутри SwiftUI View.

## 1. App Control Workspace

### Взаимодействие

Новый экран или рабочая область принимает drag-and-drop:

- `.app` bundle из Finder;
- ярлык запущенного приложения из списка Processes;
- сохранённую карточку ранее наблюдавшегося приложения.

После добавления появляется карточка приложения с текущим состоянием и явной кнопкой начала сессии. Само перетаскивание не должно автоматически завершать процесс, менять priority или запускать длительный сбор данных.

### Что измерять в сессии

- длительность наблюдения;
- средний, peak и p95 CPU;
- средний и peak RSS;
- число процессов и дерево потомков;
- disk read/write deltas, если метрика доступна для PID;
- threads, wakeups и energy impact, если их можно получить стабильным публичным или допустимым локальным API;
- корреляцию с memory pressure и thermal state всего Mac;
- события launch, exit, появление helper/child process и sustained-load interval;
- baseline в покое и отклонение от предыдущих сессий.

Thermal state и общесистемная сеть не должны автоматически приписываться выбранному приложению. Такие данные показываются как одновременные системные события, пока не появится надёжная per-process атрибуция.

### Действия пользователя

Безопасный первый набор:

- открыть приложение или показать его в Finder;
- перейти к Process Detail;
- включить локальное уведомление о sustained load;
- завершить процесс после подтверждения;
- принудительно завершить только отдельным действием с повторным предупреждением;
- остановить наблюдение, очистить историю или экспортировать выбранную сессию.

Автоматический kill, скрытый `SIGSTOP`, постоянное изменение priority и обещание «ограничить CPU» не должны входить в первую версию. Для них нет универсального безопасного контракта, а приостановка приложения может привести к потере данных или зависанию связанных процессов.

### Хранение аналитики

Данные остаются локальными. Пользователь должен видеть:

- включена ли история;
- срок хранения и максимальный размер;
- какие метрики недоступны из-за прав;
- кнопку полного удаления истории;
- экспорт одной сессии в JSON/CSV без содержимого файлов, командных секретов и сетевого трафика.

## 2. Расширяемый каталог AI-агентов

Поддержку агентов следует оформить как registry адаптеров. Для каждого адаптера фиксируются:

- стабильный ID и отображаемое имя;
- install anchors и известные executable names;
- допустимые command-line признаки;
- конфигурационные roots;
- способы обнаружения MCP, skills, plugins и локальных indexes;
- тестовые fixtures для защиты от ложного совпадения;
- уровень поддержки: `process only`, `components`, `usage limits`, `arena compatible`.

Новые агенты не должны добавляться одной строкой с общим keyword: слишком широкое совпадение может ошибочно приписать MacCleaner чужой процесс.

## 3. Проверка лимитов и потребления

Нельзя достоверно вычислить остаток облачного лимита только по локальному процессу. Источник данных должен показываться рядом со значением:

- официальный provider API;
- локальный CLI с документированным machine-readable output;
- локальный usage-файл агента, если его формат стабилен и пользователь разрешил чтение;
- ручное значение пользователя;
- `Недоступно`, если проверяемого источника нет.

Токены, стоимость, reset time и rate-limit headers следует хранить раздельно. Авторизационные данные должны использовать Keychain или уже авторизованный официальный инструмент; MacCleaner не должен извлекать секреты из command line, логов или интерфейса стороннего приложения.

## 4. Native Agent Arena

Arena нужна не для абстрактного рейтинга моделей, а для повторяемого сравнения агентов на одном Mac.

Минимальный сценарий:

1. пользователь выбирает агентов и тестовый набор;
2. каждый агент получает одинаковую версию входных данных в отдельной временной рабочей области;
3. MacCleaner фиксирует start/end, exit status и проверяемый результат;
4. параллельно собираются CPU, peak RSS, child processes, длительность и доступные usage/cost metrics;
5. результаты показываются вместе с версией агента, модели, конфигурацией и условиями запуска.

Подходящие типы тестов:

- генерация структурированного ответа по локальному fixture;
- поиск дефекта в небольшом изолированном проекте;
- исправление теста с автоматической проверкой результата;
- tool-use сценарий с разрешённым локальным mock server;
- длительный сценарий для оценки стабильности и системной стоимости.

Arena должна быть opt-in, не использовать рабочий репозиторий без явного выбора, не отправлять секреты и не считать субъективное качество доказанным без проверяемого evaluator. Сравнивать нужно одновременно корректность, время, стоимость и нагрузку на Mac.

## 5. Граф связей агента с приложениями и утилитами

Для каждого события хранится тип доказательства:

- `observed` — parent/child PID, executable path, launch/exit, открытый локальный endpoint или другой непосредственно наблюдаемый факт;
- `configured` — MCP server, tool или executable явно указан в конфигурации агента;
- `inferred` — совпадение по времени или имени без прямой связи.

Визуальное дерево может выглядеть так:

```text
Agent session
├── Agent process
│   ├── MCP server
│   │   └── Local utility
│   ├── Shell / terminal child
│   │   └── Build tool
│   └── Helper process
└── Correlated system events
    ├── CPU peak
    ├── memory-pressure change
    └── thermal-state change
```

Каждое ребро должно показывать источник, timestamp и confidence. MacCleaner не должен читать содержимое документов, перехватывать трафик или утверждать, что процесс запущен агентом, только потому что оба процесса были активны одновременно.

## 6. Атрибуция нагрузки

Нагрузка сессии агента складывается из основного процесса, подтверждённых descendants, MCP/helper процессов и явно запущенных tools. Для каждого узла и всей группы полезно показывать:

- собственные и агрегированные CPU/RSS;
- вклад в peak;
- время жизни;
- начало и конец тяжёлого интервала;
- неизвестную или разделяемую нагрузку отдельной категорией.

Если сторонняя утилита уже работала до сессии агента или используется несколькими приложениями, её ресурсы нельзя целиком приписывать одному агенту.

## Предлагаемый порядок

1. Стабилизировать registry агентов и модель evidence.
2. Реализовать короткие opt-in сессии наблюдения без постоянной истории.
3. Добавить локальное хранение и сравнение с baseline.
4. Построить event/dependency graph на parent-child и configuration evidence.
5. Подключать provider limits только для подтверждённых источников.
6. После этого запускать ограниченную Arena с локальными fixtures.

## Критерии готовности к реализации

- определены публичные или допустимые системные источники каждой метрики;
- описаны permission states и fallback `Недоступно`;
- есть retention/privacy policy для истории;
- тесты различают `observed`, `configured` и `inferred`;
- terminate/force-terminate остаются явными и подтверждаемыми;
- Arena имеет изолированный workspace и воспроизводимый evaluator;
- ни один лимит или вклад в нагрузку не показывается как точный без проверяемого источника.
