# Возможности улучшения MacCleaner

Это единственный реестр планируемых, проверяемых и выполненных улучшений приложения. Новые пункты следует добавлять отдельными строками, сохраняя ссылку на отчёт, тест или другое основание. Параллельные backlog-файлы вести не следует.

## Правила ведения

- Каждый пункт получает неизменяемый ID формата `IMP-XXX`.
- Вывод без подтверждающего файла в реестр не добавляется.
- Выполненные пункты не удаляются: их статус меняется на `Выполнено`, а рядом добавляется ссылка на проверку результата.
- Если измерение не подтверждает улучшение, это также фиксируется в связанном отчёте.

### Внешние ориентиры

[Vorssaint Utils](https://github.com/vorssaint/vorssaint-utils) уже реализует близкий набор локальных macOS-инструментов. Использовать репозиторий для изучения UX, optional permission lifecycle, local-first обработки, модульного включения функций и восстановления системного состояния. Код и визуальную идентичность не копировать без отдельной проверки GPL-3.0-or-later и trademark-ограничений; MacCleaner сохраняет собственную архитектуру и minimum target macOS 13. Подробное сопоставление находится в [анализе Utilities и menu bar](../utility-tools-and-menu-bar-analysis.md#внешний-референс-vorssaint-utils).

[Project Nullframe](https://project-nullframe.vercel.app/) и [исходный репозиторий](https://github.com/m1ckc3s/nullframe) использовать как визуальный и performance-референс для `IMP-007`, `IMP-036`, `IMP-037` и `IMP-054`: модульные telemetry cards, честные `LIVE/SIM` labels, единый update loop, пониженная частота публикации snapshot, полная пауза скрытого интерфейса и пропуск offscreen render. Nothing-style и browser simulated telemetry не копировать буквально; MacCleaner сохраняет нативный macOS-стиль и показывает `Недоступно`, когда системная метрика не подтверждена. Подробное сопоставление находится в [анализе Utilities и menu bar](../utility-tools-and-menu-bar-analysis.md#внешний-референс-project-nullframe).

## Критерий решения по Pro

Предлагаемый gate для одинакового тестового corpus:

- менее 10–15% дополнительных корректных reclaimable bytes при стоимости более 3x по времени или I/O — не делать Thorough обычной пользовательской функцией;
- более 20% дополнительных корректных bytes при приемлемой стоимости — показывать как явный Deep/Pro scan;
- между этими границами — оставить опциональным режимом и продолжить измерения на целевых Mac.

Порог является продуктовой гипотезой. До парного benchmark он не считается подтверждённой характеристикой приложения.
